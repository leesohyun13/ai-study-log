# 🍳 "다 됐어요"를 믿지 않기 — 구현 후 단계별 검증 게이트 (Review Gates)

- **작성자 / 직무:** 안현호 / 개발자(안드로이드)
- **날짜:** 2026-06-17
- **상태:** ✅ 성공 (실제 운영 중)

> 🔵 **개발자용.** 원리("완료 선언 ≠ 완료")는 모든 AI 산출물 검수에 통한다(6번).
> 전체 지도 → [[00-overview]] · 작업 *전* 관문은 → [[02-brainstorm-gate]]

---

**1. 문제 (Problem)**
> AI는 **"완료했습니다"를 너무 쉽게** 말한다. 그런데 실제로는—
> - 컴파일조차 안 되거나,
> - 요청 안 한 걸 끼워 넣었거나(over-engineering),
> - 돌려보면 그냥 안 된다.
> 핵심: **"다 됐다"는 신호는 검증의 *끝*이 아니라 *시작*이다.** AI(또는 사람)의 "읽어보니 맞아요"는 증거가 아니다.

**2. 원래 걸리던 시간**
> 완료를 믿고 넘어갔다가 나중에 터져서 되짚는 비용. 늦게 터질수록 비싸다.

**3. 쓴 도구 · 프롬프트 (Recipe)**

### 원리 (누구나)

완료를 선언하기 전에 **단계별 관문**을 통과시킨다. 규모에 맞게 일부만 써도 된다.

1. **요청 충족 검증** — 요청한 게 다 들어갔나 / 안 시킨 걸 끼워넣진 않았나 / 기존 게 안 깨졌나.
   - ⭐ **기계 검증 필수**: 빌드/컴파일/테스트를 **실제로 돌려서** 에러 0을 확인한다. *"코드를 읽어보니 문제없어 보임"은 검증으로 인정하지 않는다.*
2. **실제 동작 검증** — 시나리오대로 **직접 실행**해서 눈으로 확인. (체크리스트: "X 하면 Y가 되는가" 한 줄씩)
3. **품질 검증** — 컨벤션 일관성 / 성능·메모리 / 보안. 기존 코드 패턴과 어긋나지 않나.

> ⭐ **작성자 ≠ 검수자.** 짠 사람(또는 짠 AI)이 자기 작업을 **자기가 승인하지 않는다.** 검수는 *별도 패스*로 — 만든 맥락과 분리돼야 빈틈이 보인다.

### 구현 (Claude Code 전제)

- 검수를 **별도 리뷰 에이전트**에게 위임(구현 에이전트와 분리). 절차서는 `workflows/review/`에 온디맨드로.
- "완료" 신호는 *재검증 트리거*로 취급 — 서브에이전트가 `DONE` 하면 그게 게이트의 시작.
- **무한 재작업 방지**: 같은 항목을 N회(예: 3회) 고쳐도 안 되면 멈추고 **사람에게 결정 요청**.

**4. 줄어든 시간 / 효과**
> - "완료라더니 안 됨"이 확 줄었다.
> - 기계 검증을 게이트 1의 입장 조건으로 박으니 **거짓 완료**가 통과하지 못한다.

**5. 함정 (안 되던 것 / 주의점)** ← 제일 중요
> - **"읽어봤더니 맞음"을 검증으로 침.** → 실제 빌드/실행 결과로만 인정. (파이프 출력 가공도 주의 — 종료코드 가려질 수 있음)
> - **자기가 자기 작업을 승인(self-approve).** → 작성과 검수를 같은 컨텍스트에서 하지 않는다. 별도 패스/에이전트.
> - **사소한 일에도 3단 풀가동.** → 1줄 수정엔 과하다. 규모에 맞춰 단계 생략.
> - **무한 재작업 루프.** → N회 실패 시 사람에게 에스컬레이션.
> - **요청 범위 밖 "개선"을 검수 중 슬쩍 추가.** → 그건 새 요청. [[02-brainstorm-gate]]로 다시.

**6. 재사용 가능성**
> - **비개발 동일**: AI가 만든 카피/분석/문서를 "다 됐다"로 받으면, *체크리스트로 실제 확인 + 다른 눈으로 검수* 후에만 인정. "스스로 OK"는 믿지 않는다.
> - **도구 무관**: 핵심은 두 줄 — ① 완료는 증거(실행 결과)로 / ② 작성자와 검수자를 분리.
