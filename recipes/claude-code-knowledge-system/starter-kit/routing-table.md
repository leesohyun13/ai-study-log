# "어디에 적지?" 빠른 분기표

> 적립할 게 생겼을 때 어느 자리에 넣을지 1초 만에 정하는 표.
> `docs/wiki/README.md`에 통합해 두면, `/learn`이 컴파일할 때도 이 표를 따른다.

| 항목 성격 | 자리 |
|---|---|
| 코드/작업 안티패턴 (틀림→맞음) | `gotchas.md` |
| 화면·기능 스코프 결정 (왜 이 동작을 택했나) | `decisions.md` |
| 아키텍처 결정 (모듈 경계·구조·핵심 데이터 흐름) | `docs/adr/ADR-NNN-*.md` |
| 화면·기능 자체의 상태/동작 명세 | `spec/` (→ 03-spec-at-a-glance, 2차) |
| 재사용 가능한 다단계 절차가 한 줄로 안 담길 때 | `decisions.md` 요약 + 실제 코드 경로 |
| 워크플로우 / 프로세스 규칙 | `workflows/` 류 절차서 |
| 다음 세션에도 **강제**해야 할 규칙 | `.claude/rules/*.md` 또는 `CLAUDE.md` (상시 로드) |

## 판단 기준 한 줄

- 안티패턴이 **코드 레벨** → `gotchas.md` / **프로세스 레벨** → workflow 파일
- 결정이 **한 화면** → `decisions.md` / **모듈·레이어를 가로지름** → ADR
- 성격이 **"지식(참조)"** → `docs/` (온디맨드) / **"규칙(강제)"** → rules·CLAUDE.md (상시 로드, 토큰 비용)

## decisions vs ADR 경계

ADR은 아래 중 **3개 이상** 해당할 때만:
- 여러 대안 중 하나를 선택했다
- 되돌리기 어렵거나 비용이 크다
- 6개월 후 팀원이 이유를 알아야 한다
- 코드/모듈 구조에 영향을 준다
- 다른 합리적 개발자가 다르게 결정할 수 있는 비자명한 선택이다

미달이면 `decisions.md`에 한 줄.
