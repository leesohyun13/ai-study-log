# 🍳 화면·기능 스펙을 "한눈에 상태/동작" 보이게 쓰기 (spec-writer)

- **작성자 / 직무:** 안현호 / 개발자(안드로이드)
- **날짜:** 2026-06-17
- **상태:** ✅ 성공 (실제 운영 중)

> 🔵 **개발자·기획자용.** 원리는 요구사항 정의서·기획서에도 그대로 통한다(6번).
> 전체 지도 → [[00-overview]]

---

**1. 문제 (Problem)**
> 스펙을 **산문(줄글)**으로 쓰면 사람도 AI도 핵심을 못 잡는다.
> - "이 상태일 때 화면에 뭐가 보이지?"가 문단 속에 숨어서 한눈에 안 보인다.
> - 분기 조건(if A then…, if B then…)이 글에 녹아 있어 **빠뜨린 케이스**가 생긴다.
> - 아직 **안 정해진 것(Open Questions)**이 문서 여기저기 흩어져 놓친다.

**2. 원래 걸리던 시간**
> 스펙을 매번 다시 읽고 해석하는 시간 + 빠뜨린 분기에서 나오는 버그.

**3. 쓴 도구 · 프롬프트 (Recipe)**

### 원리 (누구나)

목표는 단 하나 — **읽는 사람(사람이든 AI든)이 "상태와 동작"을 한눈에 잡게.**

1. **미정 사항은 맨 위 한곳에 모은다.** `⚠️ Open Questions` 블록을 제목 바로 아래. 본문에 흩뿌리지 말 것 — 모아둬야 빠르게 스캔되고 안 놓친다.
2. **상태/분기는 산문이 아니라 표로.** 상태머신·조건 분기처럼 생긴 건 전부 표로 옮긴다.

   | 상태 | 화면 | 사용자가 할 수 있는 것 |
   |---|---|---|
   | 로딩 | 스피너 | (없음) |
   | 빈 목록 | 일러스트 + 안내문구 | "추가" 버튼 |
   | 성공 | 항목 리스트 | 항목 탭 → 상세 |
   | 에러 | 에러 메시지 | "다시 시도" |

3. **나머지는 자유.** 표로 **억지로 욱여넣지 말 것**. 목적은 형식 통일이 아니라 "한눈에 이해". 잘 정리된 줄글은 그냥 둔다.
4. **메타데이터는 머리말(frontmatter)로 분리** — 상태/담당/티켓 등은 위에 빼고, 본문은 *동작*에만 집중.

### 구현 (Claude Code 전제)

- `/spec-writer` **스킬**로 만들어 둔다. "스펙 작성", "스펙으로 정리", "spec 변환" 같은 **키워드가 나오면 자동 발동**.
- 스킬은 commands와 달리 *설명문에 맞는 상황에서 자동으로* 불린다 — 배선표 → [[00-overview]] 2번.
- 미정 사항을 발견하면 AI가 **멋대로 정하지 않고** `⚠️ Open Questions`에 올려 사람 확인을 받게 한다.

**4. 줄어든 시간 / 효과**
> - 사람·AI 둘 다 스펙 스캔 속도↑. "이 상태에서 뭐 보여?"를 표에서 즉답.
> - 분기 누락이 줄고, 미정 사항이 한곳에 모여 결정이 빨라짐.

**5. 함정 (안 되던 것 / 주의점)** ← 제일 중요
> - **모든 걸 표로 만들기.** 오히려 안 읽힌다. 표는 *상태/분기*에만. 설명형은 줄글로.
> - **Open Questions를 본문에 흩뿌리기.** 놓치고 토큰도 낭비. → 맨 위 한곳에 collect.
> - **AI가 미정 사항을 알아서 결정.** → ⚠️로 표시만 하고 결정은 사람이.
> - **메타데이터를 본문에 반복.** (`상태: …`, `담당: …`를 본문에 또 씀) → 머리말에만.

**6. 재사용 가능성**
> - **기획자에게 특히 유용**: 요구사항 정의서를 "상태 표 + 맨 위 Open Questions"로 쓰면 개발자·AI가 바로 이해한다.
> - **도구 무관**: 노션 표·구글독으로도 원리 동일. "분기는 표로, 미정은 맨 위로, 억지 표는 금지."
